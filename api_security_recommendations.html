1. 驗證與授權 (Authentication & Authorization)

  • 使用標準驗證協議： 優先採用 OAuth 2.0、OpenID Connect (OIDC) 等業界標準進行使用者和客戶端應用的驗證。
  • 強制執行最小權限原則：： 每個應用和使用者都只應被授予其執行必要功能所需的最小權限。
  • 使用 JWT (JSON Web Tokens)：： Token 中應包含必要的聲明（claims），如使用者 ID、角色和過期時間，並使用強大的簽名演算法（如 RS256）。
  • 保護敏感的 Token：： 不要在 URL 中傳遞 Token，應使用 HTTP Header (Authorization: Bearer <token>)。

2. 資料處理與加密

  • 全程使用 TLS 加密：： 所有 API 通訊都必須使用 TLS 1.2 或更高版本進行加密，以防止資料在傳輸過程中被竊聽或篡改。
  • 對敏感資料進行加密：： 對於儲存的敏感資料（如密碼、個人識別資訊），應使用強加密演算法（如 AES-256）進行加密。
  • 驗證輸入資料：： 絕不信任任何來自客戶端的輸入。對所有輸入資料進行嚴格的驗證、過濾和清理，以防止注入攻擊（如 SQL Injection, NoSQL Injection）。
  • 限制回應中暴露的資料：： 只返回客戶端確實需要的資料，避免將整個資料庫物件序列化後傳回。

3. 速率限制與資源管理

  • 實施速率限制：： 根據使用者、IP 位址或 API 金鑰，對請求頻率進行限制，以防止暴力破解和 DoS 攻擊。
  • 設定資源上限：： 對請求可能消耗的資源（如記憶體、CPU 時間）設定上限，防止惡意請求耗盡伺服器資源。
  • 合理設計分頁：： 對於返回大量資料的請求，必須實施分頁機制，並設定每頁大小的上限。

4. 錯誤處理與日誌記錄

  • 避免洩漏詳細錯誤資訊：： 不要向使用者返回詳細的錯誤堆疊追蹤或資料庫錯誤訊息。使用通用的錯誤訊息和唯一的錯誤 ID。
  • 記錄所有安全相關事件：： 詳盡記錄所有驗證成功/失敗、授權失敗、輸入驗證失敗等事件，以便進行監控和事後分析。
  • 保護日誌檔案：： 確保日誌檔案的存取權限受到嚴格控制，且不包含任何敏感資料（如密碼、Token）。
